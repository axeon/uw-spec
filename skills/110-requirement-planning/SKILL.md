# 需求规划

## 描述

将用户的想法转化为清晰的产品需求。通过启发式多层次访谈深入理解项目背景，反复验证功能模块和业务约束，再按角色与平台组合分别规划，包含业务流程图、时序图和低保真线框图。

## 核心原则

| 原则 | 说明 |
|------|------|
| 不假设，只验证 | 每一个需求点都必须经过用户确认，从不凭空推测 |
| 深入背景 | 通过 5 层 Why 挖掘真实业务痛点，而非表面需求 |
| 启发式引导 | 使用类比、场景推演、反向思考激发用户思考 |
| 反复澄清 | 功能模块至少经过概念→边界→细节 3 轮确认 |
| 用户中心 | 每个功能必须映射到具体用户故事和使用场景 |

## 触发检查

## 规划流程

### Phase 1: 需求访谈与澄清（必须首先完成）

在编写任何文档之前，必须通过多轮启发式访谈与用户充分澄清需求细节。访谈执行原则：逐轮推进、确认理解、启发式追问、记录决策、用户确认。详见 [访谈原则](references/interview-guide.md#访谈原则)。

> **各阶段详细执行方法**：1.1-1.5 每个阶段的详细提问清单、话术示例、检查清单见 [访谈指南](references/interview-guide.md)。

#### 访谈执行原则（全局要求）

| 原则 | 执行方式 |
|------|---------|
| **逐题提问** | **每次只问一个问题，等待用户回答后再问下一个，禁止一次性列出所有问题** |
| **使用选择框** | **必须使用 `AskUserQuestion` 工具呈现可点击的选择框，禁止纯文本列出选项让用户手工录入** |
| **逐轮推进** | **每轮聚焦1个层次，完成当前阶段后再进入下一阶段** |
| **一步一确认** | **每个步骤、每个模块、每个问题都需单独确认，禁止批量确认** |

> **重要**：以上原则是**全局要求**，适用于所有阶段（1.1-1.5）。详细的访谈原则、追问策略和执行方式见 [访谈指南](references/interview-guide.md#访谈原则)。

#### 提问方式规范

所有访谈问题**必须**使用 `AskUserQuestion` 工具，格式要求：

| 参数 | 说明 |
|------|------|
| question | 具体问题内容 |
| header | 问题简短标签（≤12字符） |
| options | 3-5个可点击选项，根据上下文动态生成 |
| multiSelect | 需要多选时设为 true，单选时设为 false |

`AskUserQuestion` 工具自带"Other"选项供自由录入，无需在选项中重复添加。

**示例**：

```
AskUserQuestion({
  questions: [{
    question: "为什么现在要做这个产品？",
    header: "触发因素",
    options: [
      { label: "现有方案效率低", description: "当前工具或流程体验差、效率低" },
      { label: "开拓新市场", description: "面向新用户群体或新业务线" },
      { label: "内部管理需求", description: "企业内部流程数字化、规范化" },
      { label: "竞品压力", description: "竞争对手已有类似产品" }
    ],
    multiSelect: true
  }]
})
```

#### 访谈框架（5个阶段）

**1.1 项目背景访谈（Discovery）**

目标：理解业务上下文、成功标准和约束条件。

**执行方式：逐个提问，每问一题等待回答**

6个核心问题必须逐个提出，每问完一个等待用户回答，根据回答进行必要的追问，确认理解后再问下一个。

**1.2 需求探索与头脑风暴（Brainstorming）**

目标：在理解背景后、锁定需求前，主动探索产品可能性。

| 环节 | 产出 |
|------|------|
| 用户场景扩展 | 典型+边缘场景清单 |
| 竞品功能映射 | 竞品功能对比表 |
| SCAMPER 发散 | 3-8个创新功能创意 |
| 技术赋能探索 | 技术驱动功能清单 |
| 架构能力识别 | UniWeb/SaaS 已支持能力清单（标注无需设计） |
| 功能汇总归类 | Must/Should/Could 功能清单 |

> **架构能力识别**（环节五）：基于 UniWeb + SaaS 技术栈，识别需求中已被架构覆盖的功能，标注"架构已支持，无需额外设计"。
>
> **展示格式要求**：
> - 使用标准 Markdown 表格格式
> - 表头：`| 功能 | 架构来源 | 说明 |`
> - 确保表格在独立代码块或纯文本中展示，避免与其他格式混排
>
> 详见 [访谈指南](references/interview-guide.md#架构能力识别)。


**1.3 产品架构规划（Architecture）**

用户角色（谁在用）和终端平台（用什么设备）是 **N:N 关系**。先确定角色，再为每个角色选择平台。完整的角色表、平台表、命名规则和组合示例见 [role-platform-matrix.md](../0-init/references/role-platform-matrix.md)。

分步确认，**先选角色，再为每个角色选平台**，最后汇总为项目清单：

| 步骤 | 确认项 |
|------|--------|
| 第一步 | 用户角色 |
| 第二步 | 终端平台 |
| 第三步 | 项目清单（含后端 `{项目}-app`） |
| 第四步 | 模块划分与优先级（含依赖确认） |
| 第五步 | 非功能需求 |



**1.4 需求澄清与验证（Clarification & Verification）**

**仅对前端项目执行**，后端 app 项目仅列出清单，不参与本阶段。

| 执行规则 | 说明 |
|---------|------|
| 执行顺序 | P0前端项目 → P1前端项目 |
| 执行粒度 | 一次聚焦一个项目，按模块逐一澄清 |
| 同角色复用 | 同一角色的用户故事仅澄清一次，后续项目复用 |
| 共享模块 | 跨项目共有模块仅澄清一次 |

每个模块按 **澄清→复述验证** 一体化执行：

| 步骤 | 维度 |
|------|------|
| 第一步 | Who & Why（用户故事） |
| 第二步 | What（功能边界） |
| 第三步 | How（交互细节） |
| 第四步 | 复述验证 |

#### 流程裁剪规则

根据项目复杂度自动简化访谈流程。详见 [流程裁剪规则](references/interview-guide.md#流程裁剪规则)。

**1.5 汇总确认（Final Confirmation）**

将所有访谈信息整合为完整汇总文档，向用户提出最终确认三问，全部通过后才允许进入文档编写。

#### 访谈检查清单（1.5最终门控）

所有访谈结束后，对照以下清单确认是否可以进入文档编写（共16项，示例）：

- [ ] 项目背景和触发因素已明确
- [ ] 成功指标已量化
- [ ] 目标用户和角色已定义（含用户故事）
- [ ] ...（等共16项）

**任一项未完成，不得进入文档编写阶段。**

完整清单、追问策略和示例对话见 [访谈指南](references/interview-guide.md)。

#### 访谈记录保存

**Phase 1全部完成后（1.5汇总确认通过）统一保存一份完整的访谈记录**。

保存路径、命名规则和模板详见 [访谈记录模板](references/interview-guide.md#访谈记录模板)。

### Phase 2: PRD文档编写

访谈完成且用户确认后，进入文档编写阶段。

#### 输出范围限制（重要）

**需求阶段只输出产品需求文档（PRD），不输出技术设计文档**。

| 禁止输出 | 说明 | 正确阶段 |
|---------|------|---------|
| ❌ 数据模型文档 | 如 `data-model.md` | 设计阶段 (200-database-design) |
| ❌ 数据库设计 | 如 `database-schema.md` | 设计阶段 (200-database-design) |
| ❌ API可行性评估 | 如 `api-feasibility.md` | 设计阶段 (210-java-uniweb-design) |
| ❌ 性能架构方案 | 如 `performance-architecture.md` | 设计阶段 (210-java-uniweb-design) |
| ❌ 技术选型文档 | 如 `tech-stack.md` | 设计阶段 (210-java-uniweb-design) |

**只输出以下PRD文档**：
- `prds/README.md` 项目入口
- `{角色}-{平台}/0-main.md` 项目主文档
- `{角色}-{平台}/{序号}-{模块名}.md` 模块文档

按 [PRD模板](references/templates.md) 编写，聚焦**用户需求、功能范围、业务流程**，不涉及技术实现细节。

### Phase 3: PLAN-REVIEW 循环（必须执行）

PRD编写完成后，调用 `111-requirement-review` 进行质量评审。

| 规则 | 说明 |
|------|------|
| 通过条件 | review 评分 ≥ 95分 |
| 强制退出 | 最多5轮review（round=1~5），round ≥ 6强制退出 |
| 每轮操作 | review → 修复问题 → 重新 review |

| 轮次 | 操作 | 结果判断 |
|------|------|---------|
| 1-5 | review → 修复 → review | ≥95分通过，<95分继续 |
| ≥6 | 强制退出 | 输出当前结论 |

| 优先级 | 修复要求 |
|--------|----------|
| Critical | 本轮必须全部修复 |
| Major | 本轮修复 ≥ 80% |
| Minor | 记录但不阻塞 |

循环结束时在评审报告末尾追加结论（总轮次、最终评分、状态、遗留问题）。
