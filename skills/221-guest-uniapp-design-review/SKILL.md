# 移动端原型评审

## 描述
对UniApp移动端原型进行全面评审，确保设计质量，针对移动端特性（触摸操作、屏幕尺寸、性能等）进行专项评审。

## 使用场景
| 触发条件 | 示例 |
|---------|------|
| 移动端原型完成后 | "评审移动端原型" |
| 检查App设计 | "检查App设计" |
| 移动端评审 | "移动端原型评审" |

## 角色职责
| 角色 | 职责 | 智能体 |
|------|------|--------|
| 主导 | 原型评审 | `prototype-reviewer` |
| 协作 | 需求符合度 | `product-manager` |
| 协作 | 技术可行性 | `system-architect` |
| 协作 | 移动端实现 | `js-developer` |

## 输入
| 输入项 | 来源 | 说明 |
|--------|------|------|
| 移动端原型 | `frontend/{项目名}-guest-uniapp/` | 原型设计文件 |
| 需求文档 | `requirement/prds/*` | 功能需求参考 |
| 移动端设计 | `frontend/{项目名}-guest-uniapp/README.md` | 设计规范参考 |

## 输出
| 输出项 | 位置 | 说明 |
|--------|------|------|
| 移动端原型评审报告 | `frontend/{项目名}-guest-uniapp/reviews/REVIEW-DESIGN-YYMMDDHHMM.md` | 评审结论和问题清单 |

## 流转关系
```
通过 → 进入 320-guest-uniapp-dev 移动端开发
不通过 → 返回 220-guest-uniapp-design 修改
```

## 评审维度
| 维度 | 检查要点 |
|------|---------|
| 需求符合度 | 功能覆盖、业务流程完整 |
| 移动端体验 | 触摸操作、手势支持、适配 |
| 视觉设计 | 风格统一、字体合适、间距合理 |
| 技术可行性 | 跨平台兼容、性能优化 |
| 一致性 | 组件复用、交互统一 |

## 移动端专项评审

### 触摸与手势
| 评审项 | 检查要点 |
|--------|---------|
| 点击区域 | 按钮≥44x44px，间距足够 |
| 手势操作 | 滑动、下拉刷新、左滑删除 |
| 反馈及时 | 点击态、加载态、操作反馈 |
| 防误触 | 重要操作二次确认 |

### 屏幕适配
| 评审项 | 检查要点 |
|--------|---------|
| 响应式 | 适配不同屏幕尺寸 |
| 安全区 | 适配刘海屏、圆角屏 |
| 横竖屏 | 是否支持横屏，布局适配 |
| 字体缩放 | 支持系统字体大小调整 |

### 性能体验
| 评审项 | 检查要点 |
|--------|---------|
| 页面加载 | 骨架屏、懒加载 |
| 图片优化 | 压缩、WebP、渐进加载 |
| 列表性能 | 虚拟列表、分页加载 |
| 内存管理 | 页面销毁、资源释放 |

### 原生能力
| 评审项 | 检查要点 |
|--------|---------|
| 相机/相册 | 图片选择、拍照、预览 |
| 地理位置 | 定位精度、权限处理 |
| 扫码功能 | 二维码扫描、识别 |
| 推送通知 | 本地推送、远程推送 |

## 量化通过标准

### 通过（≥95分）
| 检查项 | 标准 | 分值 |
|--------|------|------|
| 无Critical问题 | 0个 | 必须 |
| Major问题 | ≤2个 | 30分 |
| 需求符合度 | ≥95%功能覆盖 | 20分 |
| 移动端体验 | 符合移动端规范 | 25分 |
| 技术可行性 | 跨平台可实现 | 25分 |

### 有条件通过（85-94分）
- Major问题 ≤3个且有修复计划
- 无Critical问题

### 不通过（<85分）
- 存在Critical问题（如：功能缺失、严重体验问题）
- Major问题 >3个

## 评审流程

### 1. 准备阶段
- 读取原型设计文件
- 确定评审范围（用户端/商户端）
- 准备检查清单

### 2. 执行评审
按维度检查，记录问题。评审发现记录格式和评审报告结构详见 [评审报告模版](../0-init/references/review-report-template.md)。报告中需包含"移动端专项检查"扩展统计节。

**维度**: 需求符合度/移动端体验/视觉设计/技术可行性/一致性
**评审对象**: frontend/{项目名}-guest-uniapp/
**参与人员**: @prototype-reviewer @product-manager @system-architect @js-developer
**流转方向**: 通过 -> 进入项目初始化阶段；不通过 -> 返回原型设计修改

## 输出要求
**报告位置**: `frontend/{项目名}-guest-uniapp/reviews/REVIEW-DESIGN-YYMMDDHHMM.md`

**必须包含**:
- 评审信息（日期、人员、对象、端类型）
- 各维度评审结果和得分
- 移动端专项检查表
- 问题清单（含严重程度、责任人、状态）
- 量化评审结论
- 流转方向说明
- 下一步行动项
