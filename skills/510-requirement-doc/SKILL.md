# 需求文档整理

## 描述

将项目过程中的所有需求相关资料进行整理、汇总和归档，形成完整、规范的需求文档交付物。

## 使用场景

| 触发条件 | 示例 |
|---------|------|
| 项目收尾 | "整理需求文档" |
| 归档PRD | "归档产品需求资料" |
| 准备交付物 | "准备项目交付物" |

## 角色职责

| 角色 | 职责 | 智能体 |
|------|------|--------|
| 主导 | 文档整理 | `product-manager` |
| 协作 | 文档完整性确认 | `project-manager` |
| 协作 | 需求与测试对应 | `test-engineer` |

## 交付物清单

| 序号 | 文档名称 | 说明 | 格式 | 必需 |
|------|----------|------|------|------|
| 1 | 产品需求文档(PRD) | 完整的产品需求描述 | Markdown | ✅ |
| 2 | 用户故事地图 | 用户故事全景图 | Markdown | ✅ |
| 3 | 业务流程图 | 核心业务流程可视化 | Mermaid | ✅ |
| 4 | 功能清单 | 功能模块详细清单 | Markdown | ✅ |
| 5 | 需求变更记录 | 需求变更历史记录 | Markdown | ✅ |
| 6 | 验收标准汇总 | 各功能验收标准 | Markdown | ✅ |

## 整理流程

### 1. 收集资料
- 各终端需求文档
- 用户故事和验收标准
- 业务流程图和线框图
- 需求变更记录

### 2. 汇总PRD
- 按文档结构整理
- 确保内容完整准确
- 统一格式和术语

### 3. 整理用户故事地图
- 按用户角色组织
- 按优先级排序
- 标注完成状态

### 4. 归档业务流程图
- 确保图表清晰可读
- 补充必要的文字说明

### 5. 汇总验收标准
- 按功能模块组织
- 确保AC可测试
- 标注测试状态

## PLAN-REVIEW 循环（必须执行）

需求文档整理完成后，必须进入 PLAN-REVIEW 循环，确保文档质量达标。

调用技能 `511-requirement-doc-review`，评分 ≥ 95 通过，< 95 按下表修复后重新评审（最多5轮，round ≥ 6 强制退出）:

| 优先级 | 要求 |
|--------|------|
| Critical | 本轮必须全部修复，不可遗留 |
| Major | 本轮修复 ≥ 80%，剩余标注下轮计划 |
| Minor | 记录但不阻塞，按优先级排列 |

循环结束时，在评审报告末尾追加：
```markdown
## PLAN-REVIEW 循环结论
- 总轮次: {N}/5
- 最终评分: {N}分
- 状态: {通过 | 有条件通过 | 强制退出}
- 遗留问题: Critical {N}, Major {N}, Minor {N}
```

## 输出要求

**输出位置**: `requirement/`

**目录结构**:
```
requirement/
├── prds/                          # PRD产品文档
│   ├── README.md                  # PRD主文档
│   ├── CHANGELOG.md               # PRD变更历史
│   └── {用户角色}-{终端类型}/      # 按角色和终端分目录
├── interviews/                    # 需求访谈记录
└── reviews/                       # 评审报告
    └── REVIEW-PRD-YYMMDDHHMM.md   # 按时间戳命名，24小时制
```
