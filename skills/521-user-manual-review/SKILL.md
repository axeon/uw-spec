# 用户手册评审

## 描述
对用户手册、快速入门指南、FAQ等用户文档进行全面评审，确保文档易懂、完整、准确。

## 使用场景
| 触发条件 | 示例 |
|---------|------|
| 用户文档编写完成后 | "评审用户手册" |
| 检查用户文档 | "检查用户文档" |
| 用户交付物评审 | "用户文档评审" |

## 角色职责
| 角色 | 职责 | 智能体 |
|------|------|--------|
| 主导 | 文档评审 | `test-engineer` |
| 协作 | 操作步骤准确性 | `test-engineer` |
| 协作 | 功能描述准确性 | `product-manager` |
| 协作 | 界面描述准确性 | `ui-designer` |

## 输入
| 输入项 | 来源 | 说明 |
|--------|------|------|
| 用户文档 | `manual/user-manual/` | 6项用户文档 |
| 系统界面 | 实际系统 | 界面截图验证 |
| 功能清单 | `requirement/` | 功能覆盖参考 |

## 输出
| 输出项 | 位置 | 说明 |
|--------|------|------|
| 用户手册评审报告 | `manual/reviews/REVIEW-USERS-MANUAL-YYMMDDHHMM.md` | 评审结论和问题清单 |

## 流转关系
```
通过 → 进入收尾阶段完成
不通过 → 返回用户文档修改
```

## 交付物清单
| 序号 | 交付物 | 评审要点 |
|------|--------|----------|
| 1 | 用户使用手册 | 易懂性、完整性、准确性 |
| 2 | 快速入门指南 | 简洁性、可操作性 |
| 3 | 功能操作指南 | 步骤清晰、截图准确 |
| 4 | 常见问题(FAQ) | 问题覆盖、解答准确 |
| 5 | 故障排除指南 | 方案可行、步骤清晰 |
| 6 | 版本更新说明 | 变更清晰、影响明确 |

## 评审维度
| 维度 | 检查要点 |
|------|---------|
| 完整性 | 必需文档齐全、版本一致 |
| 准确性 | 功能描述一致、步骤验证、截图准确 |
| 易懂性 | 语言通俗、无歧义、有背景说明 |
| 可操作性 | 步骤具体、有预期结果、异常处理 |

## 量化通过标准

### 通过（≥95分）
| 检查项 | 标准 | 分值 |
|--------|------|------|
| 无Critical问题 | 0个 | 必须 |
| Major问题 | ≤2个 | 30分 |
| 文档完整 | 6项齐全 | 25分 |
| 内容准确 | 与系统一致 | 25分 |
| 易懂可操作 | 新手可理解 | 20分 |

### 有条件通过（85-94分）
- Major问题 ≤3个且有修复计划
- 无Critical问题

### 不通过（<85分）
- 存在Critical问题（如：关键文档缺失、严重错误）
- Major问题 >3个
- 文档完整度 <80%

## 评审流程

### 1. 准备阶段
- 获取用户文档
- 获取系统访问权限
- 准备检查清单

### 2. 执行评审
按维度检查，记录问题。评审发现记录格式和评审报告结构详见 [评审报告模版](../0-init/references/review-report-template.md)。报告中需包含"交付物检查"扩展统计节。

**维度**: 完整性/准确性/易懂性/可操作性
**评审对象**: manual/user-manual/
**参与人员**: @test-engineer @product-manager @ui-designer
**流转方向**: 通过 -> 进入收尾阶段完成；不通过 -> 返回用户文档修改

## 输出要求
**报告位置**: `manual/reviews/REVIEW-USERS-MANUAL-YYMMDDHHMM.md`

**必须包含**:
- 评审信息（日期、人员、对象）
- 各维度评审结果和得分
- 交付物检查清单
- 问题清单（含严重程度、责任人、状态）
- 量化评审结论
- 流转方向说明
- 下一步行动项
