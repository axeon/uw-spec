# Bug修复评审

## 描述
对Bug修复的代码进行评审，确保修复方案正确、修复彻底、测试覆盖充分，且不会引入新的问题。

## 使用场景
| 触发条件 | 示例 |
|---------|------|
| Bug修复开发完成后 | "评审Bug修复代码" |
| 确认修复彻底 | "确认Bug是否彻底修复" |
| 检查修复质量 | "检查修复代码质量" |

## 角色职责
| 角色 | 职责 | 智能体 |
|------|------|--------|
| 主导 | 代码评审 | `java-lead` / `js-lead` |
| 协作 | 开发工程师 | `java-developer` / `js-developer` |
| 协作 | 测试脚本评审 | `test-engineer` |

> **职责边界**：单元测试质量由开发工程师负责，评审由 lead 执行。test-engineer 专职于 API/E2E/压力/安全测试脚本评审。

## 输入
| 输入项 | 来源 | 说明 |
|--------|------|------|
| Bug修复代码 | Bug修复分支 | 修复代码 |
| 测试代码 | Bug修复分支测试目录 | 复现测试和回归测试 |
| Bug分析报告 | 缺陷管理系统 | Bug根因分析 |
| 原始Bug描述 | 缺陷管理系统 | Bug现象描述 |

## 输出
| 输出项 | 位置 | 说明 |
|--------|------|------|
| Bug修复评审报告 | `issue/reviews/REVIEW-BUGFIX-{YYMMDD}-{简述}-{HHMM}.md` | 评审结论和问题清单 |

## 流转关系
```
通过 → 合并代码并部署上线
不通过 → 返回对应修复阶段重新修复
```

## 评审维度
| 维度 | 检查要点 |
|------|---------|
| 根因理解 | 根因分析正确、修复针对根因、无潜在类似问题 |
| 修复正确性 | Bug解决、无新问题、边界处理、异常处理 |
| 代码质量 | 编码规范、修复简洁、无冗余修改、注释说明 |
| 测试覆盖 | 复现测试、修复验证、边界测试、回归测试 |
| 影响范围 | 其他功能影响、数据库变更、接口兼容性 |

## 量化通过标准

### 通过（≥95分）
| 检查项 | 标准 | 分值 |
|--------|------|------|
| 无Critical问题 | 0个 | 必须 |
| Major问题 | ≤2个 | 30分 |
| 根因理解 | 分析正确 | 25分 |
| 修复正确性 | 彻底解决 | 25分 |
| 测试覆盖 | 有复现测试 | 20分 |

### 有条件通过（85-94分）
- Major问题 ≤3个且有修复计划
- 无Critical问题

### 不通过（<85分）
- 存在Critical问题（如：修复不彻底、引入新问题）
- Major问题 >3个
- 根因分析错误

## 评审流程

### 1. 准备阶段
- 获取Bug修复代码和测试代码
- 获取Bug分析报告
- 准备检查清单

### 2. 执行评审
按维度检查，记录问题。评审发现记录格式和评审报告结构详见 [评审报告模版](../0-init/references/review-report-template.md)。

**维度**: 根因理解/修复正确性/代码质量/测试覆盖/影响范围
**评审对象**: Bug修复分支
**参与人员**: @lead @developer @test-engineer
**流转方向**: 通过 -> 合并代码并部署上线；不通过 -> 返回重新修复

## 输出要求
**报告位置**: `issue/reviews/REVIEW-BUGFIX-{YYMMDD}-{简述}-{HHMM}.md`

**必须包含**:
- 评审信息（日期、人员、对象、Bug编号）
- 各维度评审结果和得分
- 问题清单（含严重程度、责任人、状态）
- 量化评审结论
- 流转方向说明
- 下一步行动项
